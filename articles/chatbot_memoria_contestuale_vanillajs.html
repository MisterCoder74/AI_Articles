Creare un Chatbot con Memoria Contestuale in Vanilla JS

Come ho costruito VivacityGPT Online: un chatbot con memoria contestuale usando solo HTML e Vanilla JS, zero dipendenze, integrato con le API di OpenAI.

Vivacity AI Division  ·  Agosto 2023  ·  Alessandro Demontis

I. Il Progetto: Un Clone di ChatGPT Senza Framework

Quando il team di Vivacity Design Web Agency mi ha chiesto di sviluppare un chatbot interno funzionante, la mia prima reazione istintiva è stata quella di aprire npm e cercare una libreria. Mi sono fermato. Volevo dimostrare qualcosa a me stesso prima di tutto: è davvero necessario un framework per costruire un'interfaccia conversazionale moderna con memoria contestuale? La risposta, dopo qualche giorno di lavoro su VivacityGPT Online, è stata un chiaro no.

Il progetto nasce come caso di studio reale: un clone funzionale di ChatGPT, con due personalità distinte del bot e — soprattutto — la capacità di ricordare l'intera conversazione. Stack scelto: HTML puro, Vanilla JavaScript e un file CSS separato per lo stile. Nessuna dipendenza esterna, nessun bundler, nessun transpiler. Solo tre file nella stessa directory: index.html, script.js e vdgptstyle.css.

In questo articolo racconto ogni decisione tecnica, dal markup HTML fino alla gestione asincrona delle chiamate all'API di OpenAI con il modello gpt-3.5-turbo. I prerequisiti sono minimi: una conoscenza base di HTML e JavaScript è sufficiente per seguire tutto il ragionamento.

II. La Struttura HTML: Interfaccia Minima, Funzionalità Massima

La scelta architetturale più importante lato markup è stata quella di separare nettamente due aree funzionali: una navbar fissa con il campo di input dell'utente, e una sezione scrollabile per la cronologia della chat. Questo schema — comune nelle app di messaggistica — garantisce che l'input sia sempre visibile indipendentemente dalla lunghezza della conversazione.

Il campo di testo usa l'attributo onkeydown per intercettare il tasto Invio, delegando la logica alla funzione verifyEnter(). Il pulsante di invio richiama direttamente sendMessage(). Tutto il resto è gestito dinamicamente da JavaScript, inclusa la generazione dei bubble di messaggio.

Il div#chat-container è il cuore visuale dell'applicazione: è qui che JavaScript inietta dinamicamente ogni messaggio. La sezione .chathistory è quella che riceve il comportamento di scroll automatico verso il basso a ogni nuovo messaggio.

III. La Memoria Contestuale: Il Meccanismo Chiave

Come funziona il contesto nelle API di OpenAI

Questo è il punto dove molti tutorial si fermano in superficie. Le API di OpenAI per i modelli chat completion non hanno memoria nativa tra una richiesta e l'altra: ogni chiamata è stateless per definizione. Tocca a noi, lato client, costruire e mantenere il contesto conversazionale. Il meccanismo è elegantemente semplice: ad ogni richiesta inviamo l'intero storico della conversazione come array di oggetti messages, ciascuno con un role (system, user, assistant) e un content.

La memoria contestuale in VivacityGPT non è un database né un cookie: è un array JavaScript che cresce ad ogni scambio e viene inviato integralmente all'API ad ogni nuova richiesta. Più la conversazione si allunga, più token vengono consumati — ma il modello "ricorda" tutto.

Inizializzazione della memoria e system prompt

La funzione createMemory() costruisce l'array iniziale a partire da un array di messaggi di seed. Il system prompt — il messaggio con role: 'system' — è quello che definisce la personalità del bot: in questo caso, un full stack developer specializzato in codice pulito per agenzie web.

function createMemory(messages) {
    const memory = [];
    for (const msg of messages) {
        memory.push({ role: msg.role, content: msg.content });
    }
    return memory;
}

let chatMemory = createMemory([
    { role: 'system', content: "You are a full stack developer working for Vivacity Design Web Agency. You are specialized in producing code for websites and you will always provide clean and effective code. When asked for some code, you will also provide the description for different approaches to achieve the same goal." }
]);

Cambiando il contenuto del system message, si ottiene una personalità completamente diversa del bot. È così che in VivacityGPT Online ho implementato le due modalità distinte: basta inizializzare chatMemory con un system prompt differente.

IV. Le Funzioni Core: Invio, Visualizzazione e Chiamata API

sendMessage: coordinatore del flusso

La funzione sendMessage() ha un ruolo di puro coordinamento: legge l'input, lo visualizza immediatamente nell'interfaccia, delega la chiamata API e aggiorna la memoria. Il pattern async/await rende il flusso leggibile come codice sincrono pur restando non-bloccante.

async function sendMessage() {
    const inputElement = document.getElementById('user-input');
    const userInput = inputElement.value.trim();
    if (userInput !== '') {
        showMessage("Guest", userInput);
        chatMemory = await getChatGPTResponse(userInput, chatMemory);
        inputElement.value = '';
    }
}

showMessage: rendering dinamico della chat

La funzione showMessage() gestisce la creazione dei nodi DOM per ogni messaggio. Distingue tra messaggi dell'utente (Guest) e risposte del bot (VivacityGPT), applicando classi CSS diverse. Sui messaggi del bot aggiunge automaticamente un pulsante "Copia" che usa la vecchia API document.execCommand('copy') — una scelta consapevole per massimizzare la compatibilità browser senza dipendenze.

function showMessage(sender, message) {
    const chatContainer = document.getElementById('chat-container');
    const chatSection = document.querySelector('.chathistory');
    const typingIndicator = document.getElementById('typing-indicator');
    if (typingIndicator && sender === 'VivacityGPT') {
        chatContainer.removeChild(typingIndicator);
    }
    const messageElement = document.createElement('div');
    messageElement.innerText = `${sender}: ${message}`;
    if (sender === 'Guest') {
        messageElement.classList.add('user-message');
    } else if (sender === 'VivacityGPT') {
        messageElement.classList.add('chatgpt-message');
        const copyLink = document.createElement('button');
        copyLink.innerText = 'Copia';
        copyLink.style.float = 'right';
        copyLink.addEventListener('click', function (event) {
            event.preventDefault();
            const input = document.createElement('input');
            input.value = message;
            document.body.appendChild(input);
            input.select();
            document.execCommand('copy');
            document.body.removeChild(input);
        });
        messageElement.appendChild(copyLink);
    }
    chatContainer.appendChild(messageElement);
    chatSection.scrollTop = chatSection.scrollHeight;
}

getChatGPTResponse: la chiamata all'API

Questa è la funzione più densa del progetto. Mostra l'indicatore "Typing..." prima di lanciare la chiamata fetch verso l'endpoint /v1/chat/completions di OpenAI. Nel body della richiesta, l'array chatMemory viene espanso con lo spread operator e il nuovo messaggio utente viene aggiunto in coda. Un passaggio che ritengo particolarmente interessante è il filtraggio del testo in risposta: i modelli tendono a restituire blocchi di codice con backtick markdown, che in un'interfaccia HTML pura risulterebbero come testo grezzo. La regex rimuove queste formattazioni prima di visualizzare la risposta.

async function getChatGPTResponse(userInput, chatMemory = []) {
    const chatContainer = document.getElementById('chat-container');
    const typingIndicator = document.createElement('p');
    typingIndicator.id = 'typing-indicator';
    typingIndicator.textContent = 'Typing...';
    chatContainer.appendChild(typingIndicator);
    try {
        const response = await fetch('https://api.openai.com/v1/chat/completions', {
            method: 'POST',
            headers: {
                'Content-Type': 'application/json',
                'Authorization': 'Bearer placeyourkeyhere'
            },
            body: JSON.stringify({
                "model": "gpt-3.5-turbo",
                "messages": [
                    ...chatMemory,
                    {"role": "user", "content": userInput}
                ]
            })
        });
        const data = await response.json();
        const chatGPTResponse = data.choices[0].message.content.trim();
        var cleanResponse = chatGPTResponse.replace(/(```html|```css|```javascript|```php|```python)(.*?)/gs, '$2');
        cleanResponse = cleanResponse.replace(/```/g, "");
        showMessage("VivacityGPT", cleanResponse);
        chatMemory.push({ role: 'user', content: userInput });
        chatMemory.push({ role: 'assistant', content: cleanResponse });
        return chatMemory;
    } catch (error) {
        console.error(error);
    }
}

Attenzione: in questo approccio la API key di OpenAI è esposta nel codice JavaScript lato client. Per un ambiente di produzione è imperativo proxare le chiamate attraverso un backend server-side che custodisca la chiave in modo sicuro. Questo progetto è un caso di studio e una demo: non deploiamolo in produzione così com'è.

V. Riflessioni Finali e Limitazioni dell'Approccio

Il risultato finale — visibile nella demo live su vivacitydesign.net/vdgpt — è un chatbot pienamente funzionale con personalità configurabile e memoria contestuale persistente per tutta la sessione. L'intera logica applicativa occupa meno di 100 righe di JavaScript, senza una sola riga di codice di framework.

La limitazione principale di questo approccio riguarda la scalabilità del contesto: l'array chatMemory cresce in modo illimitato durante la sessione. Per conversazioni molto lunghe, si rischia di superare il context window del modello (4.096 token per gpt-3.5-turbo nella versione base) o di generare costi API elevati. Una strategia di mitigazione pratica sarebbe quella di mantenere solo gli ultimi N scambi in memoria, oppure di implementare un meccanismo di summarization periodica del contesto.

Il progetto VivacityGPT mi ha ricordato un principio che tendo a dimenticare quando lavoro in ecosistemi JavaScript moderni: la piattaforma web nativa è già straordinariamente potente. Fetch API, async/await, manipolazione DOM — tutto ciò che serve per costruire un'interfaccia conversazionale moderna esiste già nel browser, senza installare nulla. I framework sono strumenti, non prerequisiti.

Per chi volesse estendere il progetto, i prossimi passi naturali sarebbero: gestione degli errori API più robusta con feedback visivo all'utente, implementazione di uno streaming della risposta tramite ReadableStream per simulare l'effetto di digitazione progressiva, e — assolutamente necessario per la produzione — un layer backend per proteggere la chiave API. Ma come punto di partenza per capire come funziona l'integrazione con i modelli di linguaggio? Questo è esattamente il codice da leggere.
