pHTML 2.0 – Creare Universi con Custom Tag: un Esperimento con GPT-4.1-mini

Ho chiesto a GPT-4.1-mini di inventare un linguaggio HTML per creare universi: ecco cosa ha generato, dove ha brillato e dove ha mostrato i suoi limiti.

Vivacity AI Division  ·  Novembre 2025  ·  Alessandro Demontis

I. Il Punto di Partenza: Perché Chiedere a un LLM di Inventare un Linguaggio

Lavoro con VivacityGPT Complete da diversi mesi, e una delle domande che mi pongo sistematicamente è questa: fino a dove può spingersi un Large Language Model nella generazione di sistemi software integrati e coerenti? Non snippet isolati, non funzioni di esempio — ma un ecosistema completo: linguaggio, runtime, editor, documentazione. Ho quindi deciso di condurre un esperimento strutturato usando GPT-4.1-mini attraverso la piattaforma VivacityGPT Complete, chiedendogli qualcosa di deliberatamente ambizioso: inventare un linguaggio descrittivo per creare universi, implementarne il runtime tramite Web Components API, e costruire attorno a esso un ambiente di sviluppo visuale.

Il risultato si chiama pHTML 2.0 — Planet HTML — e questa è la mia analisi critica di ciò che il modello ha prodotto, di come ha ragionato sulla gerarchia dei concetti, e di dove si sono manifestati i limiti strutturali di un LLM quando lavora su sistemi di questa complessità.

II. Il Linguaggio: Sintassi, Gerarchia e Scelte Progettuali

La struttura ad albero del multiverso

Ho formulato il prompt in modo preciso: volevo che il modello progettasse una gerarchia tag dichiarativa ispirata alla sintassi HTML, capace di descrivere corpi celesti con attributi fisici reali. La prima cosa che mi ha sorpreso è stata la coerenza con cui GPT-4.1-mini ha costruito la tassonomia: dal contenitore radice <p-multiverse> fino alle lune con attributo tidal-locked, ogni livello della gerarchia rispecchia una logica astronomica genuina.

La struttura base che il modello ha prodotto è questa:

<!DOCTYPE html>
<html>
<head>
  <link rel="stylesheet" href="p_2.0_tags.css">
</head>
<body>
  <p-multiverse name="My Multiverse">
    <p-universe name="Test Universe" version="2.1">
      <p-solarsystem name="Sistema Solare" age="4.6 billion years">
        <p-star name="Sole" type="G2V" mass="1solar" temperature="5778K" luminosity="1solar"></p-star>
        <p-earth order="3" water="70%" terrain="30%" air="breathable" population-autoevolve>
          <p-moon name="Luna" distance="384400km" tidal-locked></p-moon>
        </p-earth>
        <p-mars order="4" mass="0.107earth" radius="3389km" air="thin">
          <p-moon name="Phobos" distance="9376km"></p-moon>
          <p-moon name="Deimos" distance="23463km"></p-moon>
        </p-mars>
      </p-solarsystem>
    </p-universe>
  </p-multiverse>
  <script src="p_2.0_tags.js"></script>
</body>
</html>

Oggetti avanzati e la scelta delle unità di misura

Una delle decisioni più interessanti del modello riguarda le unità di misura: il LLM ha definito autonomamente un sistema di riferimento coerente — solar mass, earth mass, AU, light-years — senza che io lo specificassi nel prompt. Questo mi ha detto qualcosa di importante sulla capacità del modello di attingere a corpus scientifici e di applicare convenzioni di dominio in modo appropriato. Interessante anche l'estensione verso oggetti astrofisici avanzati:

<!-- Oggetti avanzati: nebulosa, pulsar, quasar -->
<p-universe name="Deep Space Objects">
  <p-nebula
    name="Orion Nebula"
    type="emission"
    distance="1344 light-years"
    size="24 light-years"
    composition="hydrogen,helium,dust"
    temperature="10000K">
  </p-nebula>

  <p-pulsar
    name="PSR B1919+21"
    type="millisecond"
    period="1.3373s"
    magneticField="1e12 gauss"
    distance="2283 light-years"
    age="10 million years">
  </p-pulsar>

  <p-quasar
    name="3C 273"
    redshift="0.158"
    luminosity="4e12 solar"
    distance="2.4 billion light-years"
    massBH="886 million solar">
  </p-quasar>
</p-universe>

Insight tecnico #1 — Attributi booleani HTML nativi: GPT-4.1-mini ha gestito correttamente il pattern degli attributi booleani HTML standard: population-autoevolve (presenza = true, assenza = false) e tidal-locked seguono la specifica HTML5 per gli attributi booleani, senza inventare sintassi proprietarie tipo population-autoevolve="true". È un segnale che il modello ha interiorizzato le convenzioni del DOM, non solo la sua sintassi superficiale.

III. L'Implementazione: Web Components da Zero

Come il LLM ha strutturato il runtime JavaScript

La parte più tecnica dell'esperimento riguardava la generazione del file p_2.0_tags.js. Ho chiesto a GPT-4.1-mini di implementare ogni tag come un Custom Element via Web Components API (ES6+). Il modello ha adottato correttamente il pattern connectedCallback() per la fase di rendering, usando this.getAttribute() per leggere gli attributi dichiarativi e costruire il DOM interno. L'implementazione di <p-earth> è rappresentativa dell'approccio generale:

<!-- Utilizzo del custom element p-earth nel documento -->
<p-earth
  order="3"
  water="70%"
  terrain="30%"
  air="breathable"
  population-autoevolve>
  <p-moon name="Luna" distance="384400km" tidal-locked></p-moon>
</p-earth>

<!-- Il CSS generato per il rendering visuale -->
<style>
/* Override visuale personalizzato per p-earth */
p-earth .planet-visual {
  background: radial-gradient(circle at 30% 30%,
    rgba(255, 255, 255, 0.3),
    hsl(220, 70%, 40%)
  );
  width: 100px;
  height: 100px;
  border-radius: 50%;
  box-shadow: 0 0 20px rgba(0, 100, 255, 0.4);
}

/* Indicatore vita */
p-earth .life-indicator {
  color: #00ff88;
  font-size: 0.85em;
  font-weight: bold;
}

/* Luna con tidal-lock */
p-moon[tidal-locked] .moon-indicator::after {
  content: " 🔒";
}
</style>

Il sistema TRAPPIST-1: esopianeti e pianeti custom

Per testare la flessibilità del linguaggio ho chiesto esplicitamente un esempio con esopianeti reali. Il modello ha distinto correttamente tra tag predefiniti (i pianeti del sistema solare) e il tag generico <p-planet> per corpi celesti custom — una scelta progettuale pulita che riflette il pattern di estensibilità tipico dei linguaggi di markup ben disegnati:

<!-- Sistema TRAPPIST-1: esempio con esopianeti -->
<p-universe name="TRAPPIST-1 System" version="2.0">
  <p-solarsystem name="TRAPPIST-1" age="7.6 billion years" type="ultra-cool-dwarf">

    <p-star
      name="TRAPPIST-1"
      type="M8V"
      mass="0.089solar"
      temperature="2566K"
      luminosity="0.000525solar"
      radius="0.1192solar">
    </p-star>

    <p-planet name="TRAPPIST-1b" order="1" mass="1.017earth" radius="1.121earth" air="none"></p-planet>
    <p-planet name="TRAPPIST-1c" order="2" mass="1.156earth" radius="1.095earth" air="thin"></p-planet>
    <p-planet name="TRAPPIST-1d" order="3" mass="0.297earth" radius="0.784earth" air="thin"></p-planet>

    <!-- TRAPPIST-1e: candidato abitabile -->
    <p-planet
      name="TRAPPIST-1e"
      order="4"
      mass="0.772earth"
      radius="0.910earth"
      water="50%"
      air="breathable"
      population-autoevolve>
    </p-planet>

    <p-planet name="TRAPPIST-1f" order="5" mass="0.934earth" radius="1.045earth" water="80%" air="thin"></p-planet>

    <p-asteroidbelt
      name="Inner Belt"
      composition="silicate,carbon"
      count="~10000"
      distance="0.6AU"
      width="0.1AU">
    </p-asteroidbelt>

  </p-solarsystem>
</p-universe>

Insight tecnico #2 — Coerenza sistemica su ~800 righe: GPT-4.1-mini ha mantenuto la coerenza della gerarchia tag attraverso l'intero sistema generato, stimabile in circa 800 righe di codice tra JS, CSS e HTML di esempio. La nomenclatura degli attributi è rimasta consistente (stessa ortografia, stesse unità) in tutti i file. Questo tipo di coerenza cross-file è uno dei segnali più significativi della capacità del modello di gestire un "contratto di sistema" implicito.

IV. L'Ambiente di Sviluppo: Quando il LLM Genera Strumenti per Sé Stesso

La parte che mi ha più colpito dell'esperimento non è stata la generazione del linguaggio in sé, ma la richiesta successiva: ho chiesto a GPT-4.1-mini di progettare un ambiente di sviluppo visuale per pHTML 2.0. Il modello ha prodotto un editor a 5 tab — Visual Editor, Import JSON, Viewer, Export, pHTML Editor — con logica di esportazione/importazione in formato .phtml e JSON. È come se, dopo aver inventato un linguaggio, il LLM avesse ragionato su quale toolchain un developer si aspetterebbe di trovare. Questo mi sembra uno dei comportamenti emergenti più interessanti: la capacità di inferire le aspettative dell'utente finale a partire dalla natura dello strumento.

La vera domanda non è se GPT-4.1-mini possa generare codice funzionante — questo lo sappiamo già. La domanda è se possa generare sistemi coerenti, dove ogni componente rispetta un contratto implicito con gli altri. In questo esperimento, la risposta è stata prevalentemente sì — con una riserva importante: la coerenza è sintattica e strutturale, non fisica. pHTML 2.0 descrive universi, non li simula. Il salto dal linguaggio descrittivo al motore computazionale rimane fuori dalla portata di un singolo session prompt, e probabilmente è giusto così.

V. Limiti, Compatibilità e Cosa Ho Imparato

Dove il modello ha mostrato i suoi confini

L'esperimento ha prodotto risultati solidi, ma non privi di imperfezioni. Il manuale generato da GPT-4.1-mini marcava alcuni attributi avanzati come "in sviluppo" — in particolare il supporto alle stelle binarie tramite attributo binary. Questo è un pattern che ho osservato spesso: il modello, quando si trova a descrivere funzionalità complesse da implementare ex-novo, tende a includerle nella specifica ma a rinviarne l'implementazione reale. È una forma di technical debt implicito che un team di sviluppo umano dovrebbe riconoscere e gestire consapevolmente.

Il rendering visuale è interamente CSS-based: i gradienti radiali simulano acqua, terreno e atmosfera in modo visivamente credibile, ma non esiste fisica reale sottostante. Gli attributi come mass, temperature, luminosity sono puramente descrittivi — il sistema non calcola orbite, non simula gravitazione, non verifica la coerenza fisica dei valori inseriti. Questo non è un difetto del modello, è un limite del prompt: non ho chiesto un simulatore fisico, e il LLM ha correttamente interpretato il perimetro della richiesta.

Compatibilità e stack richiesto

La compatibilità dell'output è stata verificata dal modello stesso nella documentazione generata: Chrome/Edge (ottima), Firefox (ottima), Safari (ottima), con requisito minimo ES6+ per la Custom Elements API. Una nota tecnica importante: i Web Components richiedono contesto di pagina web servita — non funzionano aprendo un file HTML direttamente dal filesystem su alcuni browser per restrizioni CORS. È un dettaglio che il modello ha correttamente incluso nelle note di installazione.

Il giudizio finale sull'esperimento

Usando VivacityGPT Complete come ambiente di lavoro, questo ciclo di prompt mi ha richiesto circa tre sessioni di lavoro iterativo. Il risultato — un linguaggio funzionante, un runtime in JavaScript, un editor visuale, una documentazione strutturata — è qualcosa che un team di sviluppo junior avrebbe impiegato settimane a produrre. La qualità non è production-ready senza revisione umana, ma il punto di partenza generato è straordinariamente denso e coerente. GPT-4.1-mini si è dimostrato capace di tenere in testa un "contratto di sistema" complesso attraverso generazioni successive di codice — ed è proprio questa la capacità che, come ricercatore, stavo cercando di misurare.

La prossima fase dell'esperimento sarà aggiungere un layer di validazione fisica: chiedere al modello di implementare regole di coerenza astronomica (una stella di tipo M8V non può avere la luminosità del Sole, per esempio) direttamente nel runtime JavaScript. Quello sarà il vero banco di prova per capire se un LLM può generare non solo sintassi, ma semantica computazionale.
