AI Image Rebuilder: Generare Varianti di Immagini con Vision e DALL-E 3

Come combinare GPT-4o-mini Vision e DALL-E 3 in un'unica web app Vanilla JS per analizzare un'immagine e rigenerarla automaticamente.

Vivacity AI Division  ·  Ottobre 2024  ·  Alessandro Demontis

I. Il Progetto: un Workflow Multimodale in Due Atti

Quando ho iniziato a esplorare le possibilità combinate dei modelli multimodali di OpenAI, mi sono reso conto che il vero potenziale non stava nel singolo modello, ma nell'orchestrazione di più intelligenze in sequenza. AI Image Rebuilder è esattamente questo: una web app che prende un'immagine caricata dall'utente, la fa "leggere" a GPT-4o-mini tramite le sue capacità Vision, e poi passa la descrizione testuale a DALL-E 3 per generare una variante visiva completamente nuova.

Il workflow è volutamente sequenziale e utente-attivato: Upload → Analisi Vision → Generazione immagine. Tre step distinti, tre momenti di controllo. Non c'è automazione cieca: l'utente vede la descrizione prodotta dal modello Vision prima che questa diventi il prompt per la generazione. È una scelta progettuale importante, perché permette di intercettare errori o descrizioni imprecise prima di sprecare una chiamata a DALL-E 3.

Lo stack tecnico è deliberatamente minimalista: HTML5, CSS3 con Flexbox e backdrop-filter, jQuery per il solo upload AJAX, e Vanilla JS puro per tutte le chiamate API. Una scelta che riflette un principio che condivido: usare il minimo necessario, senza framework pesanti quando non servono.

II. Struttura HTML e Interfaccia Visiva

Il layout a due pannelli

L'interfaccia è costruita attorno a un concetto semplice: mostrare l'originale e il risultato affiancati, così da rendere immediato il confronto visivo. Due image-box di uguale dimensione, separate ma simmetriche, ciascuna con bordi arrotondati e un sottile effetto glassmorphism ottenuto con backdrop-filter: blur(2px).

<div class="container">
    <div class="image-box" id="original-image">
        <img id="uploaded-image" src="" alt="Uploaded Image" style="max-width:90%;display:none;">
    </div>
    <div class="image-box" id="generated-image"></div>
</div>
<div id="text-box">
    <b>Your Image analysis will appear here...</b>
    <p id="image-analysis-output"></p>
</div>
<div id="commands">
    <button type="button" onclick="generateImage();"> Rebuild the image</button>
</div>

Noto con piacere la coerenza stilistica: il text-box che conterrà l'analisi Vision occupa il 60% della larghezza e si posiziona tra le due box immagine e i controlli di generazione. È una UI narrativa: prima vedi l'immagine, poi leggi cosa ne pensa l'AI, poi ordini la rigenerazione.

Design insight: posizionare la descrizione testuale dell'AI tra l'upload e il tasto "Rebuild" non è casuale. Trasforma il testo in un elemento interattivo del flusso, non un semplice log. L'utente legge, valuta, e solo allora procede. È trasparenza algoritmica incorporata nel layout.

III. Upload e Anteprima con jQuery AJAX

FormData e feedback immediato

L'upload è l'unico punto del progetto dove entra in gioco jQuery. La scelta è pragmatica: gestire un FormData multipart con AJAX richiede poche righe con jQuery, mentre in Vanilla JS puro aggiungerebbe verbosità non necessaria. Il file viene inviato a vupload.php sul server, e in caso di successo l'anteprima viene mostrata immediatamente usando URL.createObjectURL — nessun round-trip al server per visualizzare l'immagine caricata.

$(document).ready(function () {
    $('#vuploadBtn').click(function (e) {
        e.preventDefault();
        $.ajax({
            url: 'vupload.php', type: 'POST',
            data: new FormData($('form')[0]),
            processData: false, contentType: false,
            success: function (response) {
                const data = JSON.parse(response);
                if (data.status === "success") {
                    $('#vfeedback').html("Your file was uploaded.");
                    var file = document.getElementById('vfile-input').files[0];
                    if (file) {
                        $('#uploaded-image').attr('src', URL.createObjectURL(file)).show();
                    }
                    $('#voperations button[type="button"]').attr('onclick', `submitImage("${data.url}");`);
                }
            }
        });
    });
});

Un dettaglio elegante: al completamento dell'upload, il bottone "Analyze the image" viene aggiornato dinamicamente con il suo onclick handler che include l'URL del file appena caricato. È un modo per passare il contesto senza variabili globali aggiuntive — soluzione semplice, ma efficace per un prototipo.

IV. Vision e Memoria Contestuale: GPT-4o-mini Vede

La chiamata multimodale

Qui sta il cuore tecnico del progetto. La chiamata a GPT-4o-mini non invia solo testo: il campo content del messaggio utente è un array che combina un oggetto di tipo text e uno di tipo image_url. Il modello riceve simultaneamente l'istruzione e l'immagine da analizzare. È la struttura che abilita le capacità Vision dell'API.

body: JSON.stringify({
    model: "gpt-4o-mini",
    messages: [
        ...chatMemory.map((item) => ({ ...item })),
        { role: "user", content: [
            { type: "text", text: "Provide a detailed description of the image content. Don't thank, greet or propose the user to ask further questions, only provide the image description." },
            { type: "image_url", image_url: { url: fileUrl } }
        ]}
    ],
    max_tokens: 450
})

La scelta di GPT-4o-mini invece di GPT-4o per la Vision è economicamente razionale: per generare una descrizione testuale di un'immagine, la versione mini offre qualità più che sufficiente a un costo significativamente inferiore. Il sistema di memoria chatMemory è mutuato dal pattern di VivacityGPT: mantiene il contesto tra le chiamate, aggiungendo sia l'URL dell'immagine che la risposta del modello come coppie user/assistant.

Un'attenzione che apprezzo: la risposta viene ripulita dai marcatori Markdown bold con una regex prima di essere visualizzata nel DOM. Piccolo dettaglio, ma evita che l'utente veda asterischi nell'output.

La vera sfida del workflow multimodale non è tecnica, è epistemica: stiamo usando la descrizione che un modello fa di un'immagine come prompt per un altro modello che ne creerà una nuova. Il risultato non è una copia — è l'interpretazione di un'interpretazione. Ogni passaggio introduce una trasformazione semantica. È distillazione visiva attraverso il linguaggio.

V. DALL-E 3 Genera: Dal Testo alla Nuova Immagine

La chiamata di generazione e il download

L'ultima fase del workflow legge il testo prodotto da Vision direttamente dal DOM — document.getElementById("image-analysis-output").textContent — e lo usa come prompt per DALL-E 3. La generazione avviene a 1024×1024 pixel, l'unica dimensione quadrata disponibile per questo modello. Il risultato viene iniettato dinamicamente nella seconda image-box, accompagnato da un link di download che si apre in nuova scheda.

async function generateImage() {
    const apikey = localStorage.getItem("openaikey");
    const prompt = document.getElementById("image-analysis-output").textContent;
    const response = await fetch("https://api.openai.com/v1/images/generations", {
        method: "POST",
        headers: { "Content-Type": "application/json", Authorization: "Bearer " + apikey },
        body: JSON.stringify({
            model: "dall-e-3",
            prompt: prompt,
            n: 1,
            size: "1024x1024"
        })
    });
    const data = await response.json();
    const imageUrl = data.data[0].url;
    const image = document.createElement("img");
    image.src = imageUrl;
    const downloadLink = document.createElement("a");
    downloadLink.href = imageUrl;
    downloadLink.target = "_blank";
    downloadLink.textContent = "Download Image";
    chatContainer.appendChild(image);
    chatContainer.appendChild(document.createElement("br"));
    chatContainer.appendChild(downloadLink);
}

Sicurezza dell'API Key e considerazioni per la produzione

L'API key viene recuperata da localStorage.getItem("openaikey"). Questa strategia è perfetta per prototipi e demo personali — non espone la chiave nel codice sorgente né in un repository — ma non è adatta alla produzione, dove le chiamate alle API OpenAI dovrebbero transitare sempre attraverso un backend proxy autenticato. È una nota che vale la pena tenere sempre presente quando si scala da prototipo a prodotto.

Pattern riutilizzabile: l'architettura Vision → testo → generazione è un template applicabile a decine di use case: variazioni stilistiche di product photo, ricostruzione di bozze architettoniche, generazione di concept art da riferimenti fotografici. Il codice di questo progetto è un punto di partenza solido per qualsiasi pipeline image-to-image mediata dal linguaggio.

In sintesi, AI Image Rebuilder dimostra come sia possibile costruire un'applicazione multimodale funzionante con poche centinaia di righe di codice, nessun framework pesante, e due endpoint API. La complessità vera non è nel codice — è nel capire come far dialogare modelli con capacità complementari, dove uno osserva e l'altro immagina.
