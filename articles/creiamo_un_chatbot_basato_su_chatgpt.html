PILLOLE DI AI

Creiamo un chatbot basato su ChatGPT

Costruisci il tuo chatbot personalizzato con le API di OpenAI usando solo HTML, CSS e JavaScript: niente server, niente dipendenze.

Vivacity AI Division  ·  Aprile 2024  ·  Alessandro Demontis

I. Perché costruire il proprio chatbot invece di usare ChatGPT?

La domanda è legittima: ChatGPT è già disponibile, gratuito nella sua versione base, e funziona egregiamente. Eppure, sempre più sviluppatori — e semplici appassionati — sentono l'esigenza di avere un'istanza propria, personalizzabile, integrabile in un sito web o in un workflow specifico. Un chatbot proprietario significa controllo totale sul system prompt, sul modello utilizzato, sull'interfaccia e sulla memoria conversazionale.

In questo articolo vi guidiamo nella costruzione di un chatbot funzionante basato sulle API di OpenAI, con una particolarità importante rispetto alla maggior parte dei tutorial disponibili online: tutto il codice è client-side, scritto in puri HTML, CSS e JavaScript. Nessun server, nessuna dipendenza da Node.js, Python o yarn/npm. Funziona nel browser, in locale o su qualsiasi hosting standard.

💡 Punto chiave: Un'implementazione client-side abbassa drasticamente la barriera d'ingresso. Chiunque sappia caricare un file HTML su un server può avere il proprio chatbot operativo in meno di un'ora.

II. Architettura e principi operativi

I quattro pilastri del progetto

Il chatbot si fonda su quattro componenti logici distinti, ciascuno con un ruolo preciso all'interno del flusso conversazionale:

  • Struttura HTML: un campo di input per il prompt utente e un contenitore div inizialmente vuoto, destinato a essere popolato dinamicamente con i messaggi della chat.
  • Funzione di cattura input: un blocco JavaScript che intercetta l'invio del messaggio, lo valida e lo prepara per la trasmissione all'API.
  • Memoria conversazionale: un array — comunemente chiamato chatMemory — che accumula i messaggi scambiati, preservando il contesto della conversazione nel tempo.
  • Funzione di interrogazione del modello: il cuore del sistema, una chiamata asincrona alle API REST di OpenAI che invia il contesto e riceve la risposta generata dal modello.

Il nodo critico: la chiamata API

Tra i quattro elementi, la funzione di interrogazione è senza dubbio la più delicata. Si tratta di una richiesta HTTP POST all'endpoint ufficiale di OpenAI, autenticata tramite una API Key personale. La struttura minima della richiesta è la seguente:

const response = await fetch('https://api.openai.com/v1/chat/completions', {
    method: 'POST',
    headers: {
        'Content-Type': 'application/json',
        'Authorization': 'Bearer YOUR_API_KEY_HERE'
    },
    body: JSON.stringify({
        "model": "gpt-3.5-turbo-16k",
        "messages": [
            ...chatMemory,
            {"role": "user", "content": userInput}
        ]
    })
});

Due parametri meritano attenzione particolare: il valore Authorization, che deve contenere la vostra chiave API valida ottenuta dal pannello di OpenAI, e il campo model, che deve corrispondere esattamente al nome del modello così come riportato nella documentazione ufficiale. Un errore di ortografia in quest'ultimo campo restituisce un errore silenzioso difficile da diagnosticare.

⚠️ Osservazione editoriale: Incorporare una API Key direttamente nel codice JavaScript client-side è una pratica che espone la chiave a chiunque apra i DevTools del browser. Per un uso personale o in ambiente controllato è accettabile; per un deployment pubblico è necessario interporsi con un layer backend — anche minimo — che funga da proxy autenticato. La semplicità ha un prezzo in termini di sicurezza.

III. Costi, modelli e sostenibilità economica

Un aspetto che i tutorial entusiasti spesso sottovalutano è il modello di pricing di OpenAI. La API Key è ottenibile gratuitamente, ma ogni interrogazione al modello consuma token — unità di misura del testo elaborato, sia in input che in output — e questi token hanno un costo.

Per un uso personale o sperimentale, i costi sono contenuti: pochi dollari al mese coprono migliaia di conversazioni con modelli come GPT-3.5 Turbo. La situazione cambia se si scala verso modelli più capaci come GPT-4o, dove il costo per milione di token è significativamente più alto. È indispensabile aprire un account prepagato su OpenAI, impostare un limite di spesa mensile e monitorare il consumo dalla dashboard.

📊 Dato pratico: Con un credito iniziale di 5€ su GPT-3.5 Turbo, un utente medio può sostenere centinaia di conversazioni. Il vero rischio economico emerge solo in scenari ad alto volume o con modelli premium senza limiti configurati.

IV. Dalla risposta API all'interfaccia utente

Parsing e manipolazione della risposta

La risposta restituita dall'API di OpenAI è un oggetto JSON strutturato. Il testo generato dal modello si trova all'interno del percorso response.choices[0].message.content. Una volta estratto, questo testo può essere filtrato, formattato e iniettato dinamicamente nel DOM dell'interfaccia HTML.

Un aspetto spesso trascurato è la gestione del Markdown: i modelli di OpenAI tendono a rispondere con formattazione Markdown (grassetti, elenchi, blocchi di codice). Senza un parser dedicato, queste stringhe appaiono all'utente con i simboli grezzi, compromettendo la leggibilità. Integrare una libreria leggera come marked.js risolve il problema con poche righe di codice.

La memoria conversazionale come differenziatore

La vera intelligenza percepita di un chatbot non risiede nel modello — quello è lo stesso per tutti — ma nella gestione del contesto conversazionale. Accumulare correttamente i messaggi nell'array chatMemory, includendo sia i turni utente (role: "user") sia le risposte del modello (role: "assistant"), consente al chatbot di "ricordare" ciò che è stato detto e di rispondere in modo coerente.

🔍 Riflessione finale: Costruire un chatbot client-side con le API di OpenAI è oggi alla portata di chiunque abbia basi di JavaScript. La vera complessità non è tecnica ma progettuale: definire un buon system prompt, gestire la memoria in modo efficiente e proteggere la chiave API sono le sfide che separano un prototipo da uno strumento realmente utile. La democratizzazione dell'AI generativa passa anche da queste piccole, consapevoli costruzioni artigianali.
